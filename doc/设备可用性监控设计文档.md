# 设备可用性监控设计文档（实现版）

## 1. 文档目标

本文档描述 Home Assistant 自定义集成 `device_availability_monitor` 的当前实现方式，作为代码、README 和后续扩展的统一参考。

本版已按当前代码实现校准，覆盖 `manifest.json` 中的版本 `0.1.2`。

本版重点解决以下问题：

- 扫描并发覆盖
- 高频事件导致的重复处理与性能问题
- 设备误判，尤其是核心实体与非核心实体混杂场景
- 设备健康状态从二态升级为三态
- 抖动设备识别
- Snapshot 全量重建造成的性能浪费
- 启动或 registry 更新风暴期间传感器公开数值反复归零

当前实现适用的 Home Assistant 最低版本为 `2026.4.0`。

---

## 2. 设计目标

### 2.1 功能目标

- 按设备维度统计离线状态
- 提供 `online`、`degraded`、`offline` 三态
- 识别低电量设备
- 识别抖动设备
- 支持设备、实体、集成和 domain 排除
- 提供适合 Lovelace 和自动化消费的诊断型传感器
- 关键运行时状态可跨重启恢复，避免离线计时和 flap 统计断裂

### 2.2 工程目标

- 所有逻辑 async-safe
- 不阻塞事件循环
- `state_changed` 以单设备增量更新为主，避免事件触发全量重建
- 常规状态事件不做全局扫描；启动、`reset_stats` 和 registry 重建允许重建索引并全量扫描
- 通过 coordinator 集中管理共享状态
- 避免旧扫描覆盖新扫描
- storage 读写必须异步且可延迟合并
- pending 状态事件按 `entity_id` 去重合并；当前 `MAX_PENDING_EVENTS` 是调试日志提示阈值，不是硬性截断上限
- 启动、`reset_stats` 和 registry 重建扫描期间不发布中间统计结果；公开传感器只展示上一份完整快照或最终扫描结果

## 4. 运行时数据模型

### 4.1 配置对象

运行时配置由默认值、`entry.data` 和 `entry.options` 合并得到，后写入的 options 会覆盖初始 data。运行时配置由以下字段组成：

- `warning_threshold`
- `critical_threshold`
- `offline_strategy`
- `treat_unknown_as_offline`
- `tracked_domains`
- `exclude_devices`
- `exclude_entities`
- `exclude_integrations`
- `exclude_domains`
- `ui_refresh_interval`
- `cleanup_orphan_after_hours`
- `low_battery_threshold`
- `treat_battery_unavailable_unknown_as_low`

### 4.2 核心结构

#### `MonitoredEntity`

用于描述一个被纳入监控的实体，包含：

- `device_id`
- `entity_domain`
- `integration`
- `is_core`

#### `DeviceState`

用于描述设备的运行时状态，包含：

- `device_id`
- `device_name`
- `integration`
- `entity_ids`
- `offline_entities`
- `offline_entity_since`
- `entity_domains`
- `battery_entity_values`
- `battery_percent`
- `source_offline_entity_id`
- `source_battery_entity_id`
- `offline_since`
- `flap_history`（`deque(maxlen=20)`）
- `flap_count`
- `health_state`
- `health_reasons`

### 4.3 索引

coordinator 维护以下索引：

- `entity_index`
- `battery_entity_index`
- `_device_states`
- `last_recovered_at`

其中：

- `entity_index` 是离线监控通道，只纳入 domain 位于 `tracked_domains` 且未被排除的实体
- `battery_entity_index` 是电量监控通道，按 entity registry 中 `device_class` 或 `original_device_class` 为 `battery` 的实体识别
- `battery_entity_index` 不依赖 `tracked_domains`，也不受 `exclude_domains` 影响
- `exclude_devices`、`exclude_entities` 和 `exclude_integrations` 会同时影响离线通道和电量通道
- `exclude_domains` 只关闭对应 domain 的离线统计，不会排除同 domain 下的电量实体
- `device_state.entity_ids` 是被当前集成纳入的离线实体和电量实体合集，不是该设备在 HA registry 中的全部实体

### 4.4 持久化状态

coordinator 使用 Home Assistant `Store` 持久化以下运行时字段：

- `offline_since`
- `offline_entity_since`
- `flap_history`
- `last_recovered_at`
- `last_snapshot`

存储 key 采用 config entry 维度隔离，不改变现有 `entity_id`：

- `device_availability_monitor.<entry_id>.runtime`

存储结构如下：

```json
{
  "devices": {
    "device_id": {
      "offline_since": "2026-04-30T09:00:00+00:00",
      "offline_entity_since": {
        "light.living_room": "2026-04-30T09:00:00+00:00"
      },
      "flap_history": [
        "2026-04-30T09:01:00+00:00",
        "2026-04-30T09:03:00+00:00"
      ]
    }
  },
  "last_recovered_at": {
    "device_id": "2026-04-30T10:00:00+00:00"
  },
  "last_snapshot": {
    "offline_devices": [],
    "critical_devices": [],
    "degraded_devices": [],
    "low_battery_devices": [],
    "flapping_devices": [],
    "by_integration": {},
    "offline_warning_count": 0,
    "updated_at": "2026-04-30T10:00:00+08:00"
  }
}
```

恢复流程如下：

1. `async_initialize()` 先从 storage 读取历史数据
2. 先恢复上一份完整公开快照 `last_snapshot`
3. `_build_indexes()` 建立实体/设备索引
4. 将持久化的 `offline_since`、`offline_entity_since`、`flap_history` 回填到设备运行态
5. 启动扫描后再按当前实体状态修正最终结果

这样可以保证：

- HA 重启后不会丢失当前离线会话的起点
- flap 统计不会因为重启被清零
- `last_recovered_at` 能跨重启继续保留
- 启动或 registry 重建扫描期间不会先把传感器公开计数刷新为 0
- 首次安装或执行 `reset_stats` 后没有上一份完整快照时，传感器在首轮扫描完成前可以暂时没有数值，而不是误报 0

`last_recovered_at` 只在设备从 `offline` 恢复到非 `offline` 时更新。`cleanup_orphan_after_hours` 只用于清理已经不在当前索引中的设备恢复记录；仍然存在于索引中的设备不会仅因为恢复记录过期而被删除。

## 5. 扫描机制重构

### 5.1 版本化扫描字段

coordinator 中新增：

- `self._scan_version: int = 0`
- `self._current_scan_task: asyncio.Task | None = None`

### 5.2 `start_scan`

每次启动扫描时：

1. `self._scan_version += 1`
2. 保存 `version = self._scan_version`
3. 将 `_scan_in_progress` 置为 `True`
4. 将 `_scan_processed_entities` 置为 `0`
5. 如果已经存在上一份完整公开快照，则只发布扫描进度元数据，保留旧统计结果
6. 如有旧任务未完成，先 `cancel()`
7. 使用 `hass.async_create_task(self._scan(version))`

重要约束：

- `start_scan()` 不再调用 `_reset_snapshot_state()` 清空公开 bucket
- 扫描开始时不发布 0 计数
- 首次安装或手动清空统计后，如果还没有完整快照，则扫描完成前不强制发布临时快照

### 5.3 `_scan(version)`

扫描函数会在以下位置检查版本：

- 每个批次开始前
- 每个 entity 处理前
- 每批处理并让出事件循环后
- pending 事件应用前
- 最终 snapshot 发布前

扫描过程中还需要：

- 每批结束后 `await asyncio.sleep(0)`
- 如果已有完整快照，每批结束后可以发布扫描进度；如果没有完整快照，则等待最终扫描结果
- 批量扫描阶段只更新 canonical `DeviceState`，不把中间态同步到公开 bucket
- pending 事件补应用阶段同样只更新 canonical `DeviceState`
- 扫描完成后调用 `_rebuild_visible_buckets_from_device_states()` 一次性重建公开 bucket
- 扫描完成后将 `_has_complete_snapshot` 标记为 `True`，并持久化 `last_snapshot`
- 捕获 `CancelledError` 后直接 `raise`

### 5.4 pending 队列合并

扫描进行中，状态事件不会立即应用，而是进入 pending 队列。

策略如下：

- 新增常量 `MAX_PENDING_EVENTS = 2000`
- 当 `_scan_in_progress` 为真时：
  - 使用 `dict[entity_id] = (time_fired, new_state)` 聚合最新事件
  - 同一实体只保留最后一次状态
  - 当去重后的 pending 数量等于 `MAX_PENDING_EVENTS` 时记录调试日志
  - 当前实现不会在达到该阈值后丢弃事件，也不会触发全量重扫
- 扫描结束后会按事件时间顺序补应用这批最新事件

### 5.5 设计意义

这个机制的目标是：

- 防止旧扫描结果覆盖新状态
- 通过按实体去重降低高频事件造成的 pending 增长
- 保证扫描可以安全被新扫描打断
- 避免因为队列清空而丢失最后一条设备状态
- 避免启动或 registry 更新风暴中反复发布半成品快照
- 避免自动化因传感器短暂归零而误触发

## 6. 离线判定策略

### 6.1 `offline_strategy`

新增配置项：

- `any`
- `core`
- `quorum`

默认值：`core`

### 6.2 核心实体判断

新增函数：

```python
def _is_core_domain(domain: str) -> bool:
    return domain in {
        "light",
        "switch",
        "cover",
        "fan",
        "climate",
        "lock",
        "media_player",
        "humidifier",
        "water_heater",
        "vacuum",
    }
```

### 6.3 设备离线判定

离线判定逻辑如下：

- `any`
  - 任意离线实体即可判定设备离线
- `core`
  - 只看核心实体
- `quorum`
  - 离线实体数量达到 `device_state.entity_ids` 一半及以上时判定离线
  - `device_state.entity_ids` 是当前集成纳入的离线实体和电量实体合集，不等于 HA registry 中该设备的全部实体

### 6.4 `offline_since`

`offline_since` 的计算遵循以下规则：

- 只使用“当前仍处于 offline 状态”的实体计算设备离线起点
- `core` 模式下，只看核心实体；非核心离线不会污染设备级离线起点
- `any` 和 `quorum` 模式下，所有当前离线且仍在 `entity_index` 中的实体都可参与设备级离线起点计算
- 如果所有实体都恢复在线，`offline_since` 会被清空
- 重启后优先恢复持久化的 `offline_entity_since`，然后再由当前实体状态修正

这意味着：

- 非核心实体离线不会污染核心模式的离线起点
- 设备恢复前的连续离线时长可以稳定累计
- 严重离线的累计不会因为重启被重置成 0

## 7. 设备健康模型

### 7.1 三态模型

设备状态统一映射为：

- `online`
- `degraded`
- `offline`

优先级：

1. `offline`
2. `degraded`
3. `online`

### 7.2 统一入口

```python
def _evaluate_device(device_state) -> str:
    if _is_device_offline(device_state):
        return "offline"
    if _is_device_degraded(device_state):
        return "degraded"
    return "online"
```

### 7.3 `degraded` 判定

设备未被判定为 `offline`，且满足以下任一条件时进入 `degraded`：

1. 低电量低于阈值
2. 非核心实体异常离线
3. flap 次数达到阈值

因为 `offline` 优先级最高，`offline_strategy=any` 时，非核心实体离线会直接让设备进入 `offline`，不会再单独出现在 `degraded` 列表中。

### 7.4 `flap` 机制

新增字段：

- `device.flap_history = deque(maxlen=20)`
- `device.flap_count = int`

新增常量：

- `FLAP_WINDOW_SECONDS = 300`
- `FLAP_THRESHOLD = 3`

记录函数：

```python
def _record_flap(device):
    now = utcnow()
    # deque(maxlen=20) 会自动保留最近 N 条
    device.flap_history.append(now)
    # 删除窗口外数据并刷新 flap_count
    device.flap_count = len(device.flap_history)
```

触发时机：

- 设备健康状态变化时调用 `_record_flap(device)`，变化范围包括 `online`、`degraded`、`offline` 之间的转换
- 最近 20 条 flap 记录会持久化，重启后继续参与抖动判定
- `flapping_devices_list` 收录窗口内 `flap_count > 0` 的设备
- `degraded` 中的 `flap` 原因只在 `flap_count >= FLAP_THRESHOLD` 时出现

### 7.5 设计意义

三态模型让系统可以区分：

- 真正离线的设备
- 暂时还能工作但已经亚健康的设备
- 正常设备

这比单纯的二态离线更适合大规模部署和后续告警扩展。

## 8. Snapshot 分层

### 8.1 分层结构

Snapshot 拆分为两层：

- `self._snapshot_static`
- `self._snapshot_dynamic`
- 完整公开快照状态标记 `_has_complete_snapshot`

### 8.2 static 层

static 层保存配置项和可见 bucket 的引用。bucket 内容会在单设备状态同步或可见 bucket 重建时更新，但不会在每次发布时重新构造完整结构。

- `offline_devices`
- `degraded_devices`
- `low_battery_devices`
- `flapping_devices`
- `by_integration`
- `offline_strategy`
- `low_battery_threshold`
- `treat_battery_unavailable_unknown_as_low`
- `flap_window_seconds`
- `flap_threshold`

### 8.3 dynamic 层

每次发布 snapshot 时刷新的内容：

- `offline_count`
- `critical_count`
- `warning_count`
- `degraded_count`
- `flapping_count`
- `low_battery_count`
- 各类 total
- 各类 truncated 标记
- `scan_in_progress`
- `scan_processed_entities`
- `scan_total_entities`
- `updated_at`

`scan_in_progress`、`scan_processed_entities` 和 `scan_total_entities` 可以在扫描期更新；设备数量、设备列表和 `by_integration` 只来自上一份完整公开快照或扫描完成后的新快照。

### 8.4 发布方式

最终发布时：

```python
snapshot = {**static, **dynamic}
snapshot["offline_devices"] = self._offline_devices[:MAX_EXPOSED_OFFLINE_DEVICES]
snapshot["critical_devices"] = self._critical_devices[:MAX_EXPOSED_CRITICAL_DEVICES]
snapshot["degraded_devices"] = self._degraded_devices[:MAX_EXPOSED_OFFLINE_DEVICES]
snapshot["low_battery_devices"] = self._low_battery_devices[:MAX_EXPOSED_OFFLINE_DEVICES]
snapshot["flapping_devices"] = self._flapping_devices[:MAX_EXPOSED_OFFLINE_DEVICES]
snapshot["by_integration"] = dict(self._by_integration)
return snapshot
```

0.1.2 起，扫描期间的公开发布遵循：

- 如果 `_has_complete_snapshot=True`，扫描中发布的 snapshot 会继续使用上一份完整 bucket，只更新扫描进度属性
- 如果 `_has_complete_snapshot=False`，扫描完成前不发布临时统计结果
- 扫描结束后通过 `_rebuild_visible_buckets_from_device_states()` 统一重建 bucket，然后发布新 snapshot
- 新 snapshot 会写入 storage 的 `last_snapshot`，用于下一次启动恢复

### 8.5 禁止项

禁止每次 `state_changed` 都全量 rebuild snapshot。
禁止扫描开始时清空公开 bucket 并发布 0。

这样做的原因是：

- 3000+ entity 场景下成本过高
- 大 JSON attributes 容易造成状态对象膨胀
- 事件风暴时容易阻塞主线程

当前实现仍会在以下场景重建可见 bucket：

- 扫描完成后从 canonical `_device_states` 一次性重建公开 bucket
- `reset_stats`
- registry debounce 重建
- 周期性 UI 刷新时从 `_device_states` 重建可见 bucket，以刷新离线时长、告警等级和 flap 窗口

### 9.7 限制策略

为了避免 attributes 过大，当前实现会截断暴露列表：

- 离线设备：最多 `200`
- 严重离线设备：最多 `100`
- 亚健康设备：最多 `200`
- 低电量设备：最多 `200`
- 抖动设备：最多 `200`
- 每个设备的离线实体：最多 `10`
- 每个设备的低电量实体：最多 `10`

各列表对应的 `*_total` 会保留完整数量，只是暴露给实体属性的列表内容会被截断。

## 11. 服务与 registry

### 11.1 `reset_stats`

服务 `device_availability_monitor.reset_stats` 用于：

- 重建索引
- 清空运行时缓存
- 重新扫描当前状态
- 重新计算离线、亚健康、低电量和抖动统计

### 11.2 registry 监听

entity registry 和 device registry 的变化会触发重建。

重建采用 debounce，避免短时间内频繁重建：

- `REGISTRY_REBUILD_DEBOUNCE_SECONDS = 2`
- registry 重建会保留已有运行时状态，并启动一次新的版本化扫描

### 11.3 恢复元数据

`last_recovered_at` 会在设备从 `offline` 恢复到非 `offline` 后记录恢复时间。

清理策略如下：

- 仍在当前索引中的设备会继续保留恢复时间
- 已经不在当前索引中的设备恢复记录，会在超过 `cleanup_orphan_after_hours` 后清理
- 如果当前没有任何设备索引，所有恢复记录会被清空

### 11.4 图形化控制入口

除服务外，集成还提供一个诊断型按钮：

- `button.reset_device_stats`
  - 名称：`Reset Device Stats`
  - 中文名：`重建设备统计`
  - 图标：`mdi:refresh`
  - 点击后通过 `device_availability_monitor.reset_stats` 服务执行全量重建
  - 按钮日志会提示当前 `scan_in_progress` 状态，并明确说明该操作会打断当前扫描

按钮实现遵循以下原则：

- 使用 `ButtonEntity`
- 通过 `CoordinatorEntity` 读取当前状态
- 不直接调用 coordinator 内部方法
- 按 config entry 维度实例化；当前 `manifest.json` 和 config flow 限制为单实例

## 12. 性能与稳定性

### 12.1 性能要求

1. 启动、`reset_stats` 和 registry debounce 重建允许全量扫描
2. 全量扫描必须分批执行
3. 批次之间让出事件循环
4. `state_changed` 只更新相关实体和设备，不触发全量扫描
5. 扫描期间事件只进入 pending 队列
6. pending 以 `entity_id` 去重合并，`MAX_PENDING_EVENTS` 当前仅作为日志提示阈值
7. Snapshot 不允许每次状态事件都全量重建
8. 扫描中不发布中间统计 bucket，只发布上一份完整快照与扫描进度

### 12.2 稳定性要求

1. 旧扫描不能覆盖新扫描
2. `CancelledError` 必须继续抛出
3. entity 删除、禁用、重命名不能让 coordinator 崩溃
4. 不阻塞主线程
5. 不做同步 I/O
6. 不把全局扫描作为 `state_changed` 的运行时处理策略
7. 启动或 registry 更新风暴期间不应出现传感器计数反复归零

### 12.3 运行特性

当前实现会在以下情况下保持扫描与刷新协作：

- 启动初次扫描
- 状态事件增量更新
- 周期性 UI 刷新：仅在非扫描状态且存在离线设备或窗口内 flap 设备时运行
- registry debounce 重建

启动和 registry debounce 重建都会尽量保留上一份完整公开快照。没有上一份完整快照时，传感器会等待首轮扫描完成后再给出有效数值。

这套机制的目标是让大规模设备部署仍能保持稳定。

## 13. 验收标准

以下场景需要满足：

1. 可以正常安装并加载集成
2. `offline_strategy=core` 时，核心实体离线可以正确判定设备离线
3. `offline_strategy=any` 和 `quorum` 也能正确工作
4. 启动时的旧扫描不会覆盖新状态
5. 高频事件会按 `entity_id` 合并 pending，达到 `MAX_PENDING_EVENTS` 时只记录调试日志
6. `degraded` 能在设备未离线时区分低电量、非核心异常和达到阈值的 flap
7. `flapping_devices_list` 能反映窗口内 `flap_count > 0` 的设备
8. `sensor.unavailable_devices_list`、`sensor.low_battery_devices_list`、`sensor.degraded_devices_list`、`sensor.flapping_devices_list` 均可被 Lovelace 和模板读取
9. 3000+ entity 场景下仍能保持可用
10. `reset_stats` 可重建索引并重新计算快照
11. 正常重启或 registry 重建扫描期间，传感器不应因为扫描开始而短暂归零
12. 首次安装或清空统计后，首轮扫描完成前传感器可以暂时无值，但不能发布误导性的临时 0 统计

## 14. 结论

当前实现已经把设备可用性监控从“单纯离线统计”升级为“离线 + 亚健康 + 抖动 + 低电量”的设备健康模型。

核心变化可以概括为：

- 扫描版本化
- pending 按实体去重合并
- offline strategy 可配置
- 设备健康三态化
- flap 追踪窗口化，并区分“窗口内出现 flap”和“达到 degraded 阈值”
- snapshot 分层化
- 上一份完整公开快照持久化，扫描完成后原子替换公开统计结果
- 传感器诊断化

这套设计满足 Home Assistant 大规模设备场景的稳定性要求，也为后续告警和自动化扩展留下了清晰边界。
